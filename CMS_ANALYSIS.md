# 大陆部署场景下的 CMS 选型分析

更新时间：`2026-04-15`

## 当前结论

对于这个以内容展示和线索转化为核心的 `Next.js 16` 项目，如果后续要接入 CMS，并且要求：

- 能在中国大陆稳定访问
- 可自托管
- 内容编辑门槛不要太高
- 最好支持预览或可视化编辑

建议优先级如下：

1. `Strapi 5`
2. `Payload CMS`
3. `Directus`

## 1. Strapi 5

最适合“要尽快上线，同时让非技术人员也能较顺手编辑内容”的团队。

优点：

- 可自托管，适配国内云环境更灵活
- 后台相对成熟，内容编辑学习成本较低
- 有预览工作流，适合营销站内容管理

缺点：

- 深度可视化编辑能力不如 Sanity 一类的展示模型
- 复杂页面搭建器通常仍需额外 schema 设计

适用场景：

- 团队中运营或内容人员需要频繁修改文案、案例、FAQ、视频链接
- 希望前后端职责边界更清晰

## 2. Payload CMS

更适合前端主导、重视 Next.js 深度集成和代码可控性的团队。

优点：

- 与现代 Next.js 项目协同自然
- 内容模型代码化，版本控制更清晰
- 可自托管，扩展性强

缺点：

- 对内容编辑人员不如 Strapi 友好
- 初始架构设计成本更高

适用场景：

- 团队以开发为主，愿意用工程化方式管理内容结构
- 未来可能把案例、FAQ、视频展示、SEO 元数据完全纳入统一数据层

## 3. Directus

可以用，但对这个项目不是首选。

优点：

- 自托管成熟
- 对结构化数据管理能力强

缺点：

- 更像数据平台，不是天然的网站编辑平台
- 对品牌官网场景来说，编辑体验通常没有前两者直接

## 不建议直接优先的方向

- 强 SaaS 依赖、海外节点为主的 CMS 平台
- 对中国大陆访问与登录稳定性不可控的平台

原因很直接：

- 后台加载速度
- 账号登录稳定性
- 内容预览一致性
- 合规与网络可控性

## 对本仓库的建议

当前仓库仍然是“文件内内容驱动”模式，核心文案集中在 [`lib/site-data.ts`](/D:/work/wushi-website/lib/site-data.ts:1)。

如果未来接 CMS，建议第一批迁移的内容是：

1. 首页服务模块
2. 真实案例
3. FAQ
4. 联系方式与社媒信息
5. SEO 元数据

这样可以优先解决“频繁改动内容必须改代码”的问题，而不必一开始就做复杂页面搭建器。
